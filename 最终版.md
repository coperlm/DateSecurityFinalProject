# 混合抗量子可修订区块链系统（AES+RSA+FAEST）架构深度技术分析与项目工程实践报告

基于 AES 和 RSA 的加密系统

## 一、 行业背景、项目概述与合规性驱动力

在传统分布式账本技术的演进历程中，数据不可篡改性（Immutability）一直被视为区块链的核心信任基石。然而，随着全球数据治理与隐私保护法规（如欧盟 GDPR 的“被遗忘权”）日趋严格，绝对的不可篡改性与现实合规性产生了结构性冲突。物理删除历史区块中的敏感个人数据、违法内容往往会导致哈希指针断裂与全网共识崩溃。

本项目 **DateSecurityFinalProject** 旨在解决这一矛盾，实现了一个“可修订区块链 + 加密存证 + 后量子签名”的实验系统。系统核心目标是在保证链结构可验证的前提下，支持合规场景下的数据修订（脱敏/删除），并通过签名与校验机制保证修订操作受控、可审计。

### 1.1 系统核心特性与技术栈
项目采用了高并发、内存安全的 **Rust** 语言作为核心开发语言，配合 **Axum** 异步 Web 框架构建单体 Web 服务架构，并通过 C-FFI（外部函数接口）调用本地构建的 **FAEST C 动态库**。系统的关键特征包括：
1. **加密存储**：使用 AES-256-GCM 对明文进行对称加密。
2. **密钥封装**：使用 RSA-2048 公钥加密 AES 会话密钥，形成数字信封（Digital Envelope）。
3. **身份认证**：引入 NIST PQC 第二轮候选算法 FAEST 后量子签名对关键数据进行验证。
4. **可修订链**：基于 RSA 陷门的变色龙哈希（Chameleon Hash），实现修改数据但不改变区块哈希。
5. **前端可视化**：提供静态页面的极密上传、存证浏览、审计与合规修订全过程 UI。

## 二、 系统总体架构与数据存证模型

### 2.1 单体 Web 服务架构模块划分
系统并未采用复杂的微服务，而是专注于密码学全链路的连贯性，实现了从前端 `index.html` -> HTTP API（Axum 路由） -> 业务处理器（handlers） -> 密码模块（crypto/faest_ffi） -> 链存储（chain + JSON 持久化）的清晰单体闭环：

*   **crypto.rs / faest_ffi.rs**：囊括 AES-GCM 信封、RSA 陷门、AES-CTR 压缩函数及 FAEST C-FFI 封装。
*   **chain.rs**：区块链的内存读写锁管理与 `data/chain.json` 持久化。

### 2.2 链上存证模型与数字信封
为了满足“明文不直接存链”的安全需求，区块中核心的数据载体被设计为 `EncryptedPayload`。
当用户上传文本或文件时，系统采取 **AES-256-GCM + RSA** 的数字信封流程：
1. 随机生成 32 字节 AES 会话密钥 $K$。
2. 用 AES-256-GCM 和随机 nonce 加密明文，得到密文（含 tag） `ciphertext`。
3. 用接收方（或审计方）的 RSA 公钥加密 $K$，得到 `encrypted_key`。
4. 链上仅保留 `ciphertext`、`nonce`、`encrypted_key` 的 Base64 编码及可选的元信息（MIME 等）。

## 三、 变色龙哈希底层机制与 RSA 陷门构建的密码学推演

变色龙哈希（Chameleon Hash）是支撑“可修订且不断链”的核心底层基石。系统构建了基于大整数分解问题（RSA）的变色龙陷门。

超级管理员（合规执行者）持有陷门私钥 $d$，而全网公开加密指数 $e$ 和大整数 $N$。前向哈希的计算公式为：
$$CH = H_{AES}(Data) \cdot r^e \bmod N$$
其中 $r$ 为随机选取的因子，全网节点将其记录在区块中。

当触发合规审查需将违规数据 $Data$ 替换为 $Data'$ 时，管理员利用陷门 $d$ 求解同余方程 $H_{AES}(Data) \cdot r^e \equiv H_{AES}(Data') \cdot (r')^e \bmod N$，并计算出新的因子 $r'$：
$$r' = \left( H_{AES}(Data) \cdot H_{AES}(Data')^{-1} \right)^d \cdot r \bmod N$$
只要 $(Data', r')$ 被接纳，该区块的 $CH$ 将保持不变，整个区块链拓扑结构的连贯性得以完美维持。

## 四、 AES-CTR 抗碰撞压缩函数的特定工程实现与分析

本项目的重大工程挑战与特色之一，在于**严格按要求不使用传统哈希库（如 SHA2 / CMAC）来构造上述公式中的 $H_{AES}$**。

为了将任意长度的输入转化为变色龙哈希所需的大整数，系统实现了**基于纯 AES-CTR 的伪随机流压缩函数**。其具体逻辑路径为：
1. **伪随机流变换**：使用固定的公开 Key 和 IV，通过 AES-256-CTR 模式对输入数据进行加密操作，以此作为伪随机流变换。
2. **状态折叠（Folding）**：将变长输出按 128-bit（16 字节）进行分块，并不断通过异或（XOR）等方式将状态折叠并压缩为单一的 128-bit 状态。
3. **抗扩展混入**：为了减少简单的块碰撞模式，将原始输入长度（Length）混入状态中。
4. **终态扩散**：最后进行一次额外的 AES-CTR 扩散操作，并输出为 128-bit 的大整数型（BigUint），作为 $H_{AES}(Data)$ 的最终输出输入变色龙哈希公式。

**安全瓶颈分析**：
尽管该实现满足了特殊约束并具有极高的硬件吞吐性能，但由于其最终输出被强制折叠至 128-bit 状态，依据生日悖论（Birthday Attack），其抗自然碰撞的安全性上限约为 $2^{64}$ 次 AES 操作。在长期的公有链部署中存在一定的自然碰撞隐患。但在本实验演示系统与受控联盟链场景中，该结构已足够验证可修订密码学协议的工程可行性。

## 五、 FAEST 后量子数字签名协议的工程集成与代价

为了抵御量子计算在身份伪造层面的降维打击，系统在身份认证与不可否认性逻辑层中，战略性地引入了完全基于多方计算（MPC/VOLE-in-the-Head）与对称原语的后量子签名算法 —— **FAEST**。

### 工程构建与 C-FFI 桥接
Rust 后端通过 C-FFI 技术直接桥接调用底层 FAEST C 语言动态库（`libfaest.so`）。
* **构建过程**：依赖 `meson` 和 `ninja` 编译 FAEST 源码（`meson setup libs/FAEST/builddir && ninja -C libs/FAEST/builddir`）。
* **运行约束**：系统启动必须指定运行时动态库路径（`LD_LIBRARY_PATH=libs/FAEST/builddir:$LD_LIBRARY_PATH`）。
* **功能挂载**：在生成密钥对（keygen）、上链交易签名（sign）以及合规修订权限校验（verify）三个核心生命周期节点均调用 FAEST 接口。

由于 FAEST 最小安全变体（如 FAEST-128s）的签名大小也接近 4KB，这种空间膨胀对上链容量提出了挑战，但换取了不依赖数论难题的极高长期身份安全性。

## 六、 核心业务流程：前后端交互与纵深审计闭环

本项目摈弃了阻塞式 HTTP 响应，所有涉及高强度计算（哈希、加密、后量子签名）的写入接口均采用了“**异步任务提交返回 op_id + 进度轮询**”的设计模式（Progress 模型）。

### 6.1 加密并上链流程 (Encrypt & Mine)
1. 前端调用 `/encrypt_and_mine` 提交数据。
2. 后端执行数字信封构建（AES-GCM+RSA）。
3. 对 Payload 序列化字节进行 FAEST 签名。
4. 链模块验签后，计算初始变色龙哈希（附加随机数 $r$），追加新区块并持久化至 `chain.json`。

### 6.3 合规修订与审计闭环 (Redaction Workflow)
1. **意图准备**：调用 `/redact_prepare` 生成待修改的内容（如合规要求的新状态 `Data'`）。
2. **管理员授权**：必须调用 `/admin_sign` 生成对应数据修改事件的 FAEST 管理员签名（此举实现了去中心化环境下的可审计问责制）。
3. **碰撞核验与替换**：向 `/redact` 提交新内容与管理员签名。链底座校验签名无误后，调用前文的公式计算 $r'$。
4. **验证不断链**：核对 $CH(Data, r) \equiv CH(Data', r')$。成功后物理覆写该区块内容并标示修改。

## 七、 量子霸权威胁下的安全降级演进与现有问题优化建议

### 7.1 降级演进定性分析
在面对全面通用量子计算机（具备 Shor 和 Grover 算法）的假设下：
* **对称加密层（AES-256）与 FAEST 签名**：Grover 算法会导致安全强度减半，但等效 128 位的防御力仍不可被短时间内物理击穿，身份认证层**具备极强的抗量子韧性**。
* **变色龙哈希层（RSA 陷门）**：Shor 算法可在多项式时间内分解 $N$，导致超级管理员的陷门私钥 $d$ 暴露。哈希的物理不可篡改性瓦解，意味着从结构上这属于一个**“混合过渡型方案”**。
* **兜底屏障**：由于伪造修改必须具备对应的后量子 FAEST 签名，即便黑客算出了哈希碰撞的 $r'$，也会在 `/redact` 接口验证层被拦截。系统完成了从“数据必然不可篡改”向“篡改必被阻截与溯源”的转化。

### 7.2 当前项目局限与优化建议
根据工程运行现状，项目存在以下可以进一步优化的工程与安全空间：

1. **部署门槛较高**：重度依赖本地 C-FFI 库编译及环境变量。
   * **优化建议**：推进 Docker 容器化打包，将 `libfaest.so` 及其运行依赖固化至底层镜像中。
2. **管理员绑定存在越权隐患**：当前演示模式下，管理员公钥存在由客户端传入或未做严格限制的情况。
   * **优化建议**：服务端需强绑定硬编码的管理员公钥白名单。
3. **审计追溯未完全上链**：目前的合规修订主要覆盖旧数据，虽然操作经过了签名，但审计记录未作为独立凭证。
   * **优化建议**：将管理员的修订签名与修订元数据作为“交易附加指令”显式入链，或建立旁路审计链。
4. **架构的长期演进（去 RSA 化）**：为了彻底抵御 Shor 算法，远期规划中应考虑用格密码（Lattice-based，如 LWE 问题）构造后量子变色龙陷门，并结合 Shamir 门限签名降低单点私钥暴露风险。
