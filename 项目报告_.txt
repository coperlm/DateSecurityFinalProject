1. 项目概述
本项目实现了一个“可修订区块链 + 加密存证 + 后量子签名”的实验系统，技术栈为 Rust + Axum + 本地 FAEST C 动态库。系统核心目标是：在保证链结构可验证的前提下，支持合规场景下的数据修订（例如隐私法规中的删除/脱敏要求），并通过签名与校验机制保证修订操作受控、可审计。

项目主要特征：
1）加密存储：使用 AES-256-GCM 对明文进行对称加密。
2）密钥封装：使用 RSA-2048 公钥加密 AES 会话密钥，形成数字信封。
3）身份认证：使用 FAEST 后量子签名对关键数据签名并验签。
4）可修订链：使用基于 RSA 陷门的变色龙哈希，实现“修改数据但保持区块哈希不变”。
5）前端可视化：提供上传、加密、上链、修订、校验、回读全过程 UI。

2. 需求分析与设计目标
2.1 功能需求
（1）用户可以输入文本或上传文件并完成加密上链。
（2）系统支持返回步骤级进度，便于演示和排错。
（3）管理员可对历史区块进行合规修订，且不中断哈希链。
（4）系统支持链查询、明文回读（在权限允许下）和修订结果展示。

2.2 安全需求
（1）明文不直接存链，链上只保留加密载荷和必要元数据。
（2）上链前需签名校验，防止伪造数据写入。
（3）修订需管理员签名，防止任意用户触发修订。
（4）系统需提供完整性验证逻辑，发现篡改可快速识别。

2.3 设计目标
（1）保证“可运行、可演示、可解释”。
（2）保证算法公式与代码行为一致。
（3）支持教学展示：从密码原语到业务流程全链路可观察。

3. 系统总体架构与模块划分
3.1 总体架构
系统为单体 Web 服务架构：
前端（static/index.html） -> HTTP API（Axum 路由） -> 业务处理器（handlers） -> 密码模块（crypto/faest_ffi） -> 链存储（chain + data/chain.json）。

3.2 模块划分
（1）main.rs：服务入口、密钥加载、路由注册、状态初始化。
（2）crypto.rs：AES-GCM、RSA 数字信封、FAEST 调用适配、AES-CTR 压缩函数、变色龙哈希。
（3）faest_ffi.rs：FAEST C-FFI 封装（keygen/sign/verify）。
（4）chain.rs：区块结构、上链、修订、链校验、持久化。
（5）handlers.rs：API 接口逻辑与异步任务调度。
（6）progress.rs：进度状态模型、进度查询、TTL 清理。
（7）static/index.html：前端交互与可视化展示。

3.3 运行时关键状态
（1）管理员 RSA 私钥与公钥（服务端加载/生成并持久化）。
（2）ChameleonHash 参数（由管理员 RSA 私钥推导）。
（3）区块链状态（内存读写锁 + JSON 文件持久化）。

4. 密码学方案与关键算法
4.1 数字信封流程（AES-256-GCM + RSA）
流程如下：
步骤1：随机生成 32 字节 AES 会话密钥 K。
步骤2：用 AES-256-GCM 和随机 nonce 加密明文得到 ciphertext。
步骤3：用 RSA 公钥加密 K 得到 encrypted_key。
步骤4：链上保存 ciphertext、nonce、encrypted_key（均 Base64 编码）。

优点：结合了对称加密的高性能与非对称加密的密钥分发能力。

4.2 FAEST 后量子签名
系统通过 C-FFI 调用本地 FAEST 动态库：
（1）生成密钥对。
（2）对消息签名。
（3）验签后决定是否允许上链/修订。

说明：FAEST 作为后量子签名方案，避免依赖传统离散对数/大整数分解签名假设。

4.3 变色龙哈希（基于 RSA 陷门）
前向哈希计算：
CH = H_AES(Data) * r^e mod N
其中 r 属于 Z_N*。

修订时，管理员利用陷门 d 计算 r'，使得：
H_AES(Data) * r^e ≡ H_AES(Data') * (r')^e (mod N)
并可推得：
r' = ( H_AES(Data) * H_AES(Data')^{-1} )^d * r mod N

这使得区块内容可替换但区块 hash 保持不变，实现“可修订不断链”。

4.4 普通哈希替代为纯 AES-CTR 压缩函数
项目当前实现已按要求不使用传统哈希库（如 SHA2/CMAC）来构造 H_AES。
实现逻辑：
（1）使用固定公开 key/iv 的 AES-256-CTR 对输入进行伪随机流变换。
（2）将任意长度输出折叠到 128-bit 状态。
（3）混入输入长度，减少简单碰撞模式。
（4）再做一次 AES-CTR 扩散，输出 128-bit BigUint。

这条路径实现了“基于 AES-CTR 的压缩函数”，并与变色龙哈希计算兼容。

5. 数据结构与链上存证模型
5.1 EncryptedPayload
字段包括：
（1）ciphertext：AES-GCM 密文（含 tag）
（2）nonce：GCM nonce
（3）encrypted_key：RSA 加密后的 AES 密钥
（4）filename/mime_type：文件元信息（可选）

5.2 Transaction
字段包括：
（1）tx_id：交易唯一标识
（2）payload：加密载荷
（3）sender_signature：发送者签名
（4）sender_pub_key：发送者公钥

5.3 Block
字段包括：
（1）index
（2）timestamp
（3）tx
（4）prev_hash
（5）randomness（r）
（6）hash（CH）

5.4 Blockchain
本质为有序 block 数组，支持：
（1）创世块生成
（2）新增区块
（3）合规修订
（4）链完整性校验
（5）JSON 持久化读写

6. 接口设计与前后端交互流程
6.1 核心接口
（1）GET /health：健康检查。
（2）GET /keys：返回临时 FAEST 密钥与 RSA 公钥。
（3）GET /chain：返回链数据和有效性标志。
（4）POST /encrypt_and_mine：异步提交加密上链任务，返回 op_id。
（5）GET /progress/:id：轮询进度。
（6）POST /redact_prepare：生成管理员签名消息。
（7）POST /admin_sign：管理员签名生成。
（8）POST /redact：提交合规修订。
（9）GET /block_plain/:index：读取并解密区块明文（或返回抹除标记）。

6.2 异步任务模式
关键耗时接口均采用“立即返回 op_id + 轮询进度”模式，避免前端超时并提升可观测性。

6.3 进度状态模型
每个任务由多个步骤组成，步骤状态为 queued/in_progress/done/failed，最终写入 done/success/result。

7. 关键业务流程实现
7.1 加密并上链流程
（1）接收请求并解析文本/文件数据。
（2）执行数字信封加密。
（3）对 payload 序列化字节进行 FAEST 签名。
（4）链模块验签并追加新区块。
（5）持久化 data/chain.json。
（6）回传任务结果和新增区块信息。

7.2 区块明文回读流程
（1）按索引定位区块。
（2）若为已修订区块（REDACTED 标识），直接返回抹除文本。
（3）否则执行 envelope_decrypt，返回明文内容。

7.3 合规修订流程
（1）根据 block_index + redaction_label 生成 new_content。
（2）校验管理员签名。
（3）调用链模块计算 r' 并替换 payload/randomness。
（4）验证 old/new CH 相等。
（5）持久化并返回修订结果。

7.4 创世块流程
系统启动时自动加载已有链；不存在或加载失败时创建创世块并保存。

8. 工程实现、构建运行与测试验证
8.1 构建依赖
（1）Rust toolchain
（2）meson + ninja（FAEST 库构建）
（3）可用端口 8080
（4）运行时动态库路径（LD_LIBRARY_PATH）

8.2 典型运行命令
（1）构建 FAEST：meson setup libs/FAEST/builddir && ninja -C libs/FAEST/builddir
（2）构建 Rust：cargo build（或 FAEST_BUILD_WRAPPER=1 cargo build）
（3）运行服务：LD_LIBRARY_PATH=libs/FAEST/builddir:$LD_LIBRARY_PATH ./run.sh

8.3 当前验证状态
（1）cargo check 可通过。
（2）若未设置动态库路径，运行或测试会报 libfaest.so 加载失败。
（3）设置正确路径后可正常运行联调。

8.4 测试建议
（1）单元测试：envelope roundtrip、签名验签、碰撞保持。
（2）接口测试：成功与失败路径覆盖。
（3）端到端测试：前端按钮链路自动化。
（4）性能测试：大文件与并发场景。

9. 安全性与量子场景分析
9.1 经典场景安全性
（1）机密性：由 AES-GCM 和数字信封保障。
（2）完整性：由 GCM tag + 签名 + 链校验保障。
（3）可审计性：由进度记录、链记录和修订结果展示保障。

9.2 量子场景分析
（1）AES 在量子场景下仍具备可接受安全边界（尤其 AES-256）。
（2）FAEST 作为后量子签名增强身份认证抗量子能力。
（3）RSA 在 Shor 算法下存在结构性风险，意味着变色龙陷门长期可持续性受限。

9.3 定性结论
系统属于“混合过渡型”方案：
签名与身份层具备较强后量子韧性，
但基于 RSA 的陷门部分长期需迁移到更适合的后量子可修订结构。

10. 现有问题、优化建议与后续工作
10.1 当前问题
（1）运行时依赖本地 FAEST 动态库路径，部署门槛较高。
（2）管理员身份绑定在演示模式下仍可进一步强化。
（3）部分前端文案历史残留，需要与最新 AES-CTR 压缩实现保持一致。

10.2 优化建议
（1）部署层：容器化打包，固化动态库与运行环境。
（2）安全层：服务端固定管理员公钥白名单，避免客户端任意传入。
（3）审计层：将管理员修订签名与修订元数据显式入链或单独审计链。
（4）工程层：增加 CI（fmt/clippy/test）与接口契约测试。
（5）算法层：中长期推进去 RSA 化，探索全后量子可修订机制。

10.3 后续计划
（1）短期：完善文案、补全回归测试、补充异常处理。
（2）中期：强化授权模型和审计留痕。
（3）长期：演进为全后量子版本。

11. 结论
本项目完成了“加密存证 + 可修订链 + 后量子签名”的可运行原型，具备较强教学展示价值与工程参考价值。其突出贡献在于：
（1）将合规修订需求与链式完整性验证结合。
（2）将后量子签名接入真实业务流程。
（3）将普通哈希替换为纯 AES-CTR 压缩函数，满足项目特定约束。

在未来工作中，项目可围绕“身份绑定强化、审计留痕完善、后量子全栈迁移”持续演进。
